# Intersection JSON 仕様メモ

## 読み取れた意図

`intersection_sample.json` は、交差点を単なる座標点として置くのではなく、道路とは別の独立オブジェクトとして保持し、そこから交差点形状を手続き生成するためのデータに見える。

やろうとしていることは次の通り。

- 交差点ごとに `id` を持たせる
- どの道路がその交差点に接続しているかを `connectedIds[].prim_id` で持たせる
- `type` で通常交差点とラウンドアバウトを切り替える
- `entryDist`、`outerRadius`、`islandRadius` のようなパラメータで交差点形状を生成する
- 道路側は「交差点までの線形」を持ち、交差点内部の形状は交差点オブジェクト側で管理する

つまり、今の `Road + point + start/end intersection` より一歩進めて、
「道路ネットワークの位相」と「交差点形状生成」を分けたい設計だと解釈できる。

## サンプル JSON の特徴

サンプルの各要素は次の形になっている。

```json
{
    "id": "f256151e",
    "name": "",
    "type": 1,
    "entryDist": 12.0,
    "outerRadius": 20.0,
    "islandRadius": 10.0,
    "roadCount": 1,
    "connectedIds": [
        { "prim_id": "9277cf66" },
        { "prim_id": "b72ec752" },
        { "prim_id": "b8aedeb9" },
        { "prim_id": "b260fce1" }
    ]
}
```

ここから読めること:

- `connectedIds` は接続道路の参照
- 3 本接続、4 本接続の交差点がある
- `type` に少なくとも 2 種類ある
- `outerRadius` と `islandRadius` はラウンドアバウト向けのパラメータに見える
- 通常交差点でも同じフィールドを持っているので、生成系を共通化したかった可能性が高い

## フィールド解釈

現時点の推定仕様:

| フィールド | 型 | 推定意味 |
|---|---|---|
| `id` | string | 交差点 ID |
| `name` | string | 名前。空文字可 |
| `type` | int | 交差点種別。通常交差点 / ラウンドアバウト |
| `entryDist` | float | 接続道路の終端から交差点処理開始位置までの距離 |
| `outerRadius` | float | 交差点外形、またはラウンドアバウト外周半径 |
| `islandRadius` | float | 中央島半径。ラウンドアバウトで主に使用 |
| `roadCount` | int | 交差点側の補助パラメータ。現状サンプルでは常に `1` |
| `connectedIds` | array | 接続道路 ID の配列 |
| `connectedIds[].prim_id` | string | 接続先道路の ID |

## type の解釈

これは推定だが、依頼内容とパラメータ名から見て次の対応に見える。

- `type = 0`: 通常交差点
- `type = 1`: ラウンドアバウト

ただし、サンプルだけでは `0/1` の定義は断定できない。実運用コードがあればそこで確定したい。

## このデータを現プロジェクトへ持ち込む場合の方向

この JSON を今のエディタに取り込むなら、交差点は次の段階に進むはず。

1. 交差点に `type` を持たせる
2. 交差点ごとに接続道路一覧を持てるようにする
3. 通常交差点とラウンドアバウトで別の生成ロジックを持つ
4. 道路の始終点接続だけでなく、交差点オブジェクトが道路群を束ねる
5. 将来的に交差点内の誘導線、中央島、流入部オフセット生成へつなげる

## 現在の理解

要するに、やりたいことは「道路ポリラインをつなぐための点」ではなく、
「接続道路を参照しながら交差点そのものを生成・管理するデータモデル」に移ることだと思う。
